# QoE 노멀라이저 — 영업이익 질(質) 스크리닝

회사명 하나를 넣으면 **EBITDA 브릿지**와, 영업이익 안에 파묻힌 **일회성 조정 후보**를
화면으로 올립니다. 조정은 사용자가 토글로 켜고, 일회성인지 아닌지의 **판정은 사람이** 합니다.
도구는 후보를 빠짐없이 올릴 뿐(리콜 우선), "이건 일회성이다"를 단정하지 않습니다.

M&A 실사에서 인수대상의 영업이익이 얼마나 "질 좋은" 이익인지 — 반복 가능한 본업 이익인지,
아니면 일회성·회계처리로 부풀거나 눌린 이익인지 — 를 빠르게 훑기 위한 도구입니다.

> **로컬 전용.** 서버는 이 PC 안(127.0.0.1)에서만 돕니다. 호스팅하지 않습니다.
> 실사 데이터가 PC를 떠나지 않는 것이 이 도구의 전제입니다.

## 1. 프로그램 소개

파이프라인은 4단계입니다. 웹앱은 이 검증된 파이프라인을 **그대로 호출**만 합니다(재계산
공식·게이트를 재구현하지 않습니다).

1. **재무제표 수집 (screen)** — OpenDART에서 다년 영업이익·영업활동현금흐름을 받아, 이익이
   현금으로 뒷받침되는지 괴리를 산수로 계산합니다. LLM 없음.
2. **주석 후보 발굴 (surface)** — 감사보고서·연결재무제표 주석을 Claude로 읽어, 회색지대
   후보(매출원가·판관비에 묻힌 일회성, 하단 손상·처분 등)를 **인용과 함께** 올립니다.
   같은 입력을 **3회 반복**해 회차 간 안정성을 함께 봅니다. 여기에 **상단 전용 2차 스윕**을 더합니다 —
   영업비용을 구성하는 주석(비용의 성격별 분류·매출원가·판관비, 그리고 재고자산·매출채권 손상·충당부채
   상세)만 **구조로 골라**(계정명 키워드 아님 — 섹션 총액이 매출원가·판관비 IS 라인과 산술로 맞는지, 또
   재고평가·대손·충당부채전입 같은 표준 XBRL 성분 개념코드가 있는지) 집중 발굴합니다. 전체를 한 번에
   훑는 1차가 지나치기 쉬운 덜 두드러진 항목(대손상각비·충당부채 전입 등)을 잡아 1차와 합집합합니다.
   구조로 못 고르면 그 회사는 2차를 건너뜁니다(추측 금지).
3. **D&A 추출** — EBITDA 가산분(감가상각비 + 무형자산상각비 + 리스 사용권자산 감가)을
   주석에서 결정론적으로 뽑습니다. 모든 숫자에 주석 출처를 답니다.
4. **화면 구성 (normalize)** — 위를 합쳐 단일 HTML 화면으로 만듭니다. 표시위치=상단·조정대상
   후보만 토글로 EBITDA를 움직이고, 참고(하단·불명)는 움직이지 않습니다.

### 한 페이지 통합 — ① 이익의 질 → ② 조정 EBITDA (스크롤)

이 도구엔 두 축이 있습니다. **(1) 이 회사가 보고한 영업이익을 믿을 만한가**(재무제표로 검증),
**(2) 그 이익에서 일회성을 걷어내면 얼마인가**(주석 발굴·조정). 이익 자체를 못 믿으면 일회성을
걷어내봤자 의미가 없으므로, 순서를 이익 검증부터로 둡니다.

두 축은 하나의 이야기라 **한 페이지에 세로로 담습니다** — 위에서 ① 이익의 질을 보고, 아래로
스크롤하면 ② 조정 EBITDA. 상단 **고정 네비**(회사명·종목코드·기준연도 + [① 이익의 질][② 조정
EBITDA])로 길어진 페이지를 다스리고, 감사인이 두 축을 오갈 때 로딩·스크롤 위치를 잃지 않게 합니다.

- **① 이익의 질** (재무제표 기반, 결정론): 아래 지표 6개를 LLM 없이 산수로만 계산하고 각 숫자에
  원문보기(재무제표 라인)를 답니다.
  1. **영업이익 vs 영업현금흐름** — 이익이 현금으로 뒷받침되나(발생액 = 영업이익 − 영업현금흐름).
  2. **발생액 비율** — 발생액을 총자산·매출로 나눠 규모와 무관하게.
  3. **매출채권 회전** — 채권÷매출, 회수기간(일). 안 받은 돈으로 매출을 앞세웠나.
  4. **재고 회전** — 재고÷매출원가, 재고일수. 나중에 평가손실로 터질 재고가 쌓이나.
  5. **EBITDA vs 영업현금흐름** — 우리가 계산한 EBITDA와 실제 영업현금의 차이(기준연도).
  6. **영업이익률 추이** — 영업이익÷매출. 갑자기 좋아졌/나빠졌으면 확인 대상.
  - **무엇을 판정하지 않나:** 임계·색·등급을 붙이지 않습니다(산업마다 정상 범위가 다름 — 제조는
    재고가 크고 서비스는 없습니다). 좋다/나쁘다·안전/위험을 도구가 단정하지 않고, 숫자·추이만
    정직하게 냅니다. 동종업계 비교도 하지 않고 **한 회사의 자기 추이만** 봅니다. 판정은 사람이.
- **② 조정 EBITDA**: 위 3~4단계 결과. 영업이익 안에 파묻힌 일회성 후보를 토글로 조정.

> 통합 리포트는 두 화면을 그대로 담고 있어(자기완결) 파일 하나만 열면 됩니다. 개별 페이지
> (`quality_`·`screen_`)도 그대로 남아 직접 열어볼 수 있습니다.

### surface는 왜 3회 고정인가 (사용자 선택 옵션이 없습니다)

회차 비교는 장식이 아니라 **방어의 전제**입니다. 표시위치 안정성 게이트(회차마다 상단/불명이
갈리면 조정에서 빼는 방어), 재현 배지, 리콜(회차 합집합)이 모두 3회 반복 위에 섭니다.
1회로 줄이면 이 방어들이 조용히 죽는데, 사용자는 그걸 모른 채 결과를 믿게 됩니다.
그래서 반복 횟수를 줄이는 옵션을 두지 않았습니다.

## 2. 실행법

```
1. pip install -r requirements.txt      (최초 1회 — 아래 .bat/.sh 가 자동으로 처리)
2. start_qoe.bat  더블클릭  (Windows)   /   ./start_qoe.sh  (mac·Linux)
3. 브라우저가 http://127.0.0.1:5000 로 자동으로 열립니다
4. API 키 2개 입력 (아래 발급처)  →  회사명 또는 6자리 종목코드 입력
5. 실행 전 화면이 **예상 비용·소요시간**을 먼저 보여줍니다 — 같은 공시를 이미 분석했으면
   **저장된 결과 재사용(무료·즉시)**, 아니면 [새로 분석]을 누릅니다
6. 재무제표 수집 → 주석 발굴(3회) → D&A → 화면 구성 진행 표시를 보고, 완료되면 통합 리포트
```

- **결과 재사용(무료).** 같은 회사·같은 공시(사업보고서 접수번호)를 이미 분석했으면 저장된 결과를
  그대로 재활용합니다(Claude 재호출 0 = 무료·즉시). 새 사업보고서가 올라오면 자동으로 다시 분석합니다.
  재사용은 결정론 단계(화면 구성)만 다시 도는 **완전 재활용**이라 정확도에 영향이 없습니다.

- 실행 창(콘솔)은 **열어둔 채로** 두세요. 창을 닫으면 서버가 멈춥니다.
- 종료: 콘솔 창을 닫거나 Ctrl+C.
- 금융업(은행·보험·증권)은 손익 구조가 달라 **범위 밖**입니다(아래 한계 참고).

### API 키 발급처

| 키 | 용도 | 비용 | 발급 |
|---|---|---|---|
| **OpenDART API 키** | 공시·재무제표·주석 수집 | 무료 | <https://opendart.fss.or.kr/> |
| **Anthropic API 키** | 주석 후보 발굴(Claude) | 유료·소액 | <https://console.anthropic.com/> |

키는 **브라우저 화면에서 입력**합니다. 입력한 키는 이 PC의 서버 프로세스 **메모리에만** 잠시
머물다 서버를 끄면 사라집니다. **파일·로그·산출물에 저장하지 않고**, 로컬 실행이라 PC 밖으로
나가지 않습니다. (CLI로 직접 돌리려면 `.env.example`을 `.env`로 복사해 값을 채워 쓸 수도
있습니다 — 이 경우 `.env`는 `.gitignore`로 이미 제외돼 있습니다.)

## 3. 예상 소요 시간

6개 회사를 이 웹앱으로 실제 콜드 실행해 측정한 값입니다. **주석 발굴은 1차(전체)+2차(상단 집중)**
두 스윕을 3회씩 돕니다(surface 소요 = 두 스윕 합).

| 회사 | 업종 | 주석 발굴(1차+2차, 3회) | 나머지 단계 | **합계** |
|---|---|--:|--:|--:|
| SK하이닉스 (000660) | 반도체 | 438s | ~3s | **약 7.4분** |
| 삼성전자 (005930) | 전자 | 415s | ~3s | **약 7.0분** |
| 롯데케미칼 (011170) | 화학 | 441s | ~3s | **약 7.4분** |
| 대한항공 (003490) | 항공 | 378s | ~3s | **약 6.4분** |
| 네이버 (035420) | 플랫폼 | 488s | ~3s | **약 8.2분** |
| 이마트 (139480) | 유통 | 471s | ~3s | **약 7.9분** |

- **거의 전부가 주석 발굴(2단계)**입니다. 1차(전체 주석)+2차(영업비용 구성 주석)를 각각 3회 부릅니다.
- **상단 2차 스윕이 회사당 약 2~3분을 더합니다**(1차만이던 이전 기준 ~5분 → ~6.5~8분). 2차 발췌는
  영업비용 주석만이라 작지만(대개 7~35K자), 상단 리콜을 높이려 시간을 감수한 설계입니다.
- **주석 분량과 시간은 비례하지 않습니다** — 소요는 입력 길이가 아니라 발굴·추론 출력량이 좌우합니다.
- **재무제표·D&A·화면 구성은 몇 초**로 끝납니다(DART 응답은 로컬 캐시).
- **최초 1회 콜드 스타트 비용**: 처음 실행할 때 회사코드 마스터(약 30MB)를 한 번 내려받아 캐시합니다.
- 측정 6개사 모두 **10분을 넘지 않았습니다**(최대 네이버 약 8.2분).

### 비용 — 회사당 약 $5, 재분석은 무료

주석 발굴은 Claude(Opus)를 6번(1차 3 + 2차 3) 부릅니다. 새 분석 1회는 회사당 **약 $4~$6**
(주석 분량에 비례), 소요 **약 6~9분**입니다. 실행 전 `/preview` 화면이 그 회사의 예상 비용·시간을
먼저 보여주고(실측 기준 추정 — 실제 청구는 Anthropic usage), 저장된 분석이 있으면 **재사용 시 무료**를 안내합니다.

- **프롬프트 캐싱**으로 입력 토큰을 **약 52% 절감**합니다(회차1 캐시 쓰기, 회차2·3 읽기 — 산출물 불변).
  SK하이닉스 콜드 실측: 6.9분, 입력 −51.7%.
- **같은 공시 재분석은 $0** — 저장된 결과를 재사용합니다(위 실행법 참조).
- 모델은 **Opus 고정**입니다(더 싼 모델은 발굴·리콜·태그 안정성이 측정 가능하게 낮아, 리콜을 1순위로 두는
  이 도구엔 맞지 않습니다).

### 원문 대조 검증 (독립 확인)

surface 가 "상단(영업이익 안)"으로 태그한 항목이 실제로 원문에서 영업이익 안에 있는지, **8사 데모 중 2사
(중소형 — 아이센스·코미코)에 대해 별도 에이전트가 DART 원문만 보고 독립 대조**했습니다. 결과: 도구가 상단
으로 올린 항목이 전부 원문에서 실제 영업이익 안(매출원가·판관비·급여·복리후생비 등)에 있었고 — **근거 없는
'거짓 상단' 0건**. 도구의 오류는 전부 안전한 방향(불확실하면 하단→불명으로 과보수)이었고, EBITDA 를 틀리게
하는 방향(하단/불명→상단 오태그)은 없었습니다. 이 대조에서 드러난 **중소형사 과보수**(표에는 매출원가 계상이
명확한데 인용이 스티치돼 verbatim 근거가 없어 불명 강등된 재고평가손실)를, 다시 **표 구조 근거**(표 산술
정합·재고평가 개념코드)로 살렸습니다 — 계정명 키워드가 아니라 구조로만, 거짓 상단은 0을 유지합니다.
(2사 표본이며 전수 보증은 아닙니다.)

## 5. 설계 원칙

- **하드코딩·키워드 휴리스틱 금지.** 계정명 목록으로 후보를 매칭하지 않습니다. 주석 서술을 읽고
  "비반복성" 개념으로 추론합니다. 같은 "보조금"도 매년 반복이면 정상, 코로나 일회성이면 후보 —
  계정명으론 못 가릅니다. (회계기준 표준 구조·XBRL 표준 개념코드 대조는 키워드가 아니라 보편 구조라 예외.)
- **리콜 우선 — 도구는 올리고, 판정은 사람이.** "이건 일회성이다"를 도구가 단정하지 않습니다.
  애매하면 띄웁니다. 넓히면 정밀도가 떨어지는 건 정상이고, 그 조절은 슬라이더와 사람이 감당합니다.
- **임계·색·등급으로 판정하지 않는다.** 산업마다 정상 범위가 달라 도구가 선을 그으면 틀립니다
  (제조는 재고가 크고 서비스는 없습니다). 이익의 질 지표에 "이 값은 위험/안전"을 붙이지 않고,
  빨강·초록 같은 판정색도 쓰지 않습니다(스파크라인은 단색 — 추이 모양만). 숫자와 추이를 정직하게
  보여주고, 판정은 감사인이 합니다. 동종업계 비교도 하지 않고 한 회사의 자기 추이만 봅니다.
- **모든 숫자에 원문 출처(원문보기).** 후보마다 몇 번 주석·어느 서술인지 인용을 답니다. 재무제표
  라인은 XBRL로 손익계산서 표를 재구성해 뽑은 줄을 하이라이트합니다. 인용 없는 후보는 올리지 않습니다.
- **이중계상 다중 방어.** 같은 경제사건의 "합계"와 "구성"이 둘 다 조정에 들면 이중가산입니다.
  ① 주석 표 산술로 합계=구성을 확정해 잠그고(그 외 잔여는 계산값으로 명시), ② 다른 주석에
  흩어져 산술로 못 잡는 근접쌍은 추정 잠금(해제 가능)으로, ③ D&A로 이미 가산한 상각과 겹치는
  후보는 개념코드로 조정 자격을 박탈합니다.
- **표시위치를 모르면 조정하지 않는다(보수).** EBITDA = 영업이익 + D&A는 "이미 빠진 것을 도로
  더하는" 계산이라, 영업이익 위(상단)에 표시된 항목만 조정 대상입니다. 상단인지 확실치 않으면
  불명으로 두고 조정에서 뺍니다(프롬프트→XBRL/안정성→코드 근거 게이트 3겹). 근거는 **인용에 있는
  계상 위치(verbatim)**가 1순위지만, 중소형사처럼 인용이 스티치돼 verbatim이 깨진 경우 **표 구조**
  (후보 금액이 매출원가/판관비 표에 산술 정합하거나 재고평가 개념코드에 실재)로도 상단을 인정합니다 —
  계정명 키워드가 아니라 구조로만, 표 구조가 확정하면 회차 흔들림(안정성)보다 우선합니다(XBRL 결정론과
  같은 계열). 놓친 조정 기회는 사람이 원문 확인 후 **수동 스위치**로 되살릴 수 있습니다.

## 6. 한계 (자세히는 [docs/limitations.md](docs/limitations.md))

- **금융업은 범위 밖.** 은행·보험·증권은 손익 구조가 달라 "영업이익 + D&A" 전제가 성립하지 않습니다.
- **연결재무제표 미제출 소형사는 처리 불가.** 이 도구는 연결(CFS)을 전제합니다. 종속기업이 없어 별도
  재무제표만 내는 소형사는 연결 조회가 비어 스크린 단계에서 정직하게 멈춥니다(0을 성공으로 위장 안 함).
- **운용리스(리스아웃) 감가 미포착.** 자산 임대가 본업인 업종에서 운용리스자산 감가가 기타영업손익에
  별도 계상되면 D&A 가산에서 빠져 EBITDA가 과소계상될 수 있습니다(실측: 신한지주 약 5.5%).
- **IS·CIS 분리 공시 회사는 XBRL 표시위치 결정론이 한 겹 꺼집니다**(삼성·이마트). 영업이익을
  포괄손익계산서가 아닌 손익계산서에 두는 회사는 그 한 겹이 안 돌지만, 나머지 방어(프롬프트·안정성·근거
  게이트)와 EBITDA 계산은 정상입니다.
- **매출인식 품질·회계추정은 범위 밖.** 마일리지 소멸률 가정, 감가상각 개시시점 판단 등 "숫자를
  믿을 수 있나"의 문제는 "금액을 빼는 항목"이 아니라 이 뼈대에 맞지 않습니다. 다만 이익질의 **일차
  신호**는 ① 이익의 질 섹션의 **발생액·회전율**(영업이익 vs 영업현금, 매출채권·재고 회전)이 제공합니다.
- **근접 추정(2단계)의 성격 조건은 변별력이 약합니다.** 실질 방어는 99.5% 근접 임계 하나이며,
  우연히 금액이 근접한 별개 항목은 오잠금될 수 있습니다(사용자가 [묶음 해제]로 되돌림). 총계와 구성이
  금액은 다르나 개념상 겹치는 근접 관계도 이 축이 다룹니다(정확 동일 금액의 이중가산은 별도로 완전 봉합).
- **동종업계 비교는 하지 않습니다.** 이익의 질 지표는 한 회사의 자기 시계열 추이만 봅니다 —
  업종 평균·경쟁사 대비 선을 도구가 긋지 않습니다(정상 범위 판정은 사람 몫).

이 한계들은 **자동 보정하지 않습니다**. 도구는 원값·경로·출처를 투명하게 내고, 보정과 최종
판정은 원문을 본 사람이 합니다.

> **정직하게.** 이 도구는 **1차 스크리닝**이지 완전한 체크리스트가 아닙니다. 도구가 안 올린 항목이
> 없다는 보장은 없습니다(리콜 우선으로 설계했으나 전수 보증은 아님). 리콜 수치는 AI가 넓게 뽑은 초안
> 골든셋 대비 측정이라 **진짜 리콜(원문의 모든 일회성 대비)은 미상**이며, 사람이 DART 원문으로 직접
> 대조한 것은 소형 2사뿐입니다. 최종 판정은 반드시 원문을 본 감사인이 하십시오.
