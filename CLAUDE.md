# QoE Normalizer

인수대상회사의 영업이익 질(QoE)을 스크리닝하는 도구.

## 무엇을 하는가
1. screen — 다년 누적 영업이익 vs 영업현금흐름 괴리를 산수로 계산해 "이익이 수상한가"를 먼저 거른다. LLM 없음.
2. surface — 감사보고서·주석을 LLM으로 읽어 회색지대(상단 비반복·하단 일회성·규모 큰 하단 손익) 후보를 인용·태그와 함께 빠짐없이 올린다. 반복이라거나 영업이익 아래(하단)라는 이유로 스스로 거르지 않는다.
3. normalize — 후보를 토글하면 해당 소계를 재계산한다(세금 정합 포함).

## 파이프라인
extract → screen → surface → normalize → out/
- **비용(2026-07-15)**: surface LLM 호출에 프롬프트 캐싱 기본 적용(`discover.call_model` cache=on) — system+주석 본문을 캐시 프리픽스로, 같은 스윕 회차2·3은 캐시 읽기(0.10×). SK 실측 입력 −52%(총 −34%), 산출물 불변. `--no-cache` 로 끔. 파이프라인은 기존 surface 있으면 재사용(`pipeline.run reuse_surface=기본True`, 웹앱) — 게이팅·화면만 바꿀 땐 LLM 재실행 0. surface 모델은 Opus 유지(Sonnet은 ~5× 싸나 발굴·리콜·태그 안정성 측정가능하게 낮음 — `qoe-pinned-todos` 측정).
- **실사용 재사용·비용 표시(2026-07-15)**: [A-1] 재사용은 **공시 접수번호(rcept_no) 기준 신선도**로 판정한다 — 저장된 surface 의 `source.meta.rcept_no` 와 최신 사업보고서 rcept_no 를 비교해 같으면 재사용(무료), 새 공시가 올라오면(다른 rcept) 재사용을 켰어도 자동으로 다시 분석한다(낡은 분석을 무료라고 재활용 안 함). rcept 확인 실패(오프라인)면 보수적으로 재사용 허용(`pipeline._surface_freshness`). [A-2] 실행 전 `/preview` 가 회사 해석 + 저장분 재사용 가능 여부 + **예상 비용·시간**을 먼저 보인다(LLM 안 부름=공짜). 추정은 실측 앵커(SK 106,179자→캐시 $5.04, 6~9분)를 주석 길이로 선형 스케일(`src/web/cost_estimate.py` — 표시 전용, 분석·게이트에 영향 0). **분석 자체는 안 바꾼다**(주석 무절삭·repeat 3 고정·Opus 기본): 재사용·표시만. [A-3] **빠른 스크리닝 모드(Sonnet) 설계 검토만**(구현 안 함): 두더라도 선택지로만·기본 Opus·명확한 경고("리콜 −3%p·발굴 −22%, 정밀 분석엔 Opus") 필수 — 측정상 리콜 손실이 있어 기본 전환은 리콜 1순위 원칙에 반한다. 결정: 현행 유지(Opus 기본, 캐싱·재사용으로 절감), Sonnet 토글은 사용자 요청 시 추가.

## 조정 기준선 — EBITDA
- 조정 기준선은 EBITDA = 영업이익 + (감가상각비 + 무형자산상각비 + 리스 사용권자산 감가상각비). 순이익까지 내려가지 않는다.
- 단 영업이익에 이미 차감된 상각비만 가산한다. 영업 밖에서 난 상각비는 애초에 영업이익에 안 들어 있어 되돌릴 게 없다.
- D&A는 손익계산서 본문에 안 뜨고 매출원가·판관비에 흩어진다 → 주석에서 추출한다. LLM 후보 발굴과 분리된 결정론적 숫자 추출이며 모든 숫자에 주석 출처를 단다. 리스 사용권자산 감가상각비는 성격별 주석 없는 회사에서 빠지기 쉬우니 별도 줄로 확인한다.
- 중복 방지: D&A에 이미 포함되는 정기 상각비(PPA 고객관계 상각 포함)는 조정 후보로 중복해서 올리지 않는다. D&A 단계에서 이미 되돌려지기 때문이다. **이 원칙은 프롬프트 부탁이 아니라 코드로 강제한다(2026-07-13)** — 후보 금액이 D&A 상각 셀(XBRL 개념코드 `Depreciat|Amorti` 로 식별, 비영업 `InvestmentProperty|NotInUse` 제외)과 같고 같은 주석에서 오면 조정 자격 박탈 → 구역 C 참고(`da_cell_amounts`·`_da_double_count`). 손상차손은 D&A 개념이 아니라 안 걸린다(핵심 사냥감 과제거 방지). 대한항공에서 고객관계 무형상각 99,796 이 근거 게이트 때처럼 프롬프트만으론 뚫려 구역 B 로 올라온 실사례를 코드로 막음. 상세 '조정 표시 — 이중가산 방지' 절.

## 후보의 두 태그 — 표시위치 · 조정성격
- 표시위치: 그 회사 손익계산서에서 영업이익 위(상단)냐 아래(하단)냐 — 회사가 실제로 어디 표시했는지의 사실. 확정 못 하면 불명.
- 조정성격: 일회성·비경상 경제사건에서 비롯됐나(조정대상) 아니면 시가연동·경상이라 참고냐 — 항목의 경제적 성격. 애매하면 불명.
- 영업 안/밖은 성격이 아니라 표시위치(상단/하단)로 말한다 — "비영업"이라는 성격 규정은 쓰지 않는다(폐기). 남는 성격 축은 비반복(일회성) 여부. 규모 큰 하단 손익은 "표시위치=하단 + 참고"로 표현한다(아래 범위 절).

## EBITDA 조정은 표시위치를 따른다 (배치 규칙 — 교정)
EBITDA = 영업이익 + D&A는 "영업이익에서 이미 빠진 것을 도로 더하는" 계산이다. 영업이익 아래(기타영업외 등)에 표시된 항목은 애초에 영업이익·EBITDA에 반영되지 않았으므로 조정(도로 더하기) 대상이 될 수 없다.
- 표시위치 = 영업이익 위 → 조정 가능 → 상단(조정대상, 구역 B, 토글)
- 표시위치 = 영업이익 아래 → EBITDA에 없음 → 하단(참고, 구역 C)
- 표시위치 = 불명 → EBITDA에 있는지 몰라 조정 불가 → 구역 C 불명 묶음
예: 손상차손을 영업이익 안에 표시한 회사는 조정대상(상단), 영업이익 아래(기타영업외비용)에 표시한 회사(예: SK하이닉스)는 참고(하단). "영업 아래 항목을 EBITDA에 더하는 넓은 EBITDA"는 빠지지 않은 것을 더하는 셈이라 성립하지 않는다.
- [폐기] 옛 규칙 "표시위치가 하단이어도 조정성격이 조정대상이면 상단"은 회계적으로 틀렸다 — 하단은 EBITDA에 없어 되돌릴 게 없다. surface는 성격만 태그하고, 상·하단 배치는 도구가 표시위치로 정한다.

## 표시위치 신뢰성 — 3겹 방어 (EBITDA 조정 가능 여부의 뿌리)
표시위치는 조정 가능 여부를 결정하므로 틀리면 EBITDA가 틀어진다. 그래서 표시위치는 조정성격/비반복과 **정반대로 보수적**이다(모르면 불명, 추측 금지). 방어 순서: **프롬프트(보수 규칙) → XBRL/안정성 게이트 → 코드 근거 게이트**. 프롬프트에 부탁만 하면 LLM 이 어긴다(감가상각 개시 KAM 이 근거 없이 상단 태그된 실사례) — 그래서 마지막에 코드가 강제한다.
- **[2] XBRL 결정론 확정 (`build_view.load_cis_position_lines`·`match_is_line`)**: 후보의 당기금액이 손익계산서(CIS) 라인의 당기금액과 **유일하게 일치**하면 그 라인의 표준 개념코드로 영업이익 위/아래를 결정론 확정하고 LLM 태그보다 우선한다. `IS_ABOVE_OI`/`IS_BELOW_OI`는 XBRL 표준 concept_id 집합(회계기준 구조 = no-hardcoding 예외, 한글 계정명 매칭 아님). 근거(개념코드·라인·위치)를 `표시위치_확정근거`에 남겨 원문보기에 띄운다. 집계라인을 직접 올린 후보에만 발동 — 주석에 묻힌 구성항목(재고평가·손상 등)은 집계와 금액이 달라 발동 안 함(그건 프롬프트·근거 게이트가 담당).
- **[3] 회차간 안정성 게이트 (`dedupe_candidates`가 `_disp_set` 수집 → `_merge_rows`가 union → `build_sections`)**: XBRL로 확정 안 됐는데 반복 실행에서 표시위치가 회차마다 갈리면(예: 상단/불명) 조정 대상에서 빼고 구역 C 불명 묶음으로 보낸다(`표시위치_불안정`·`표시위치_observed` 기록). 재현 배지는 "항목이 나왔나"만, 이 게이트는 "태그가 일관됐나"를 따로 본다. **단 표 구조 결정론([1c] `table_placement_ok`)이 계상 위치를 확정하면 안정성 게이트를 이긴다(2026-07-15)** — XBRL 결정론[2]이 LLM 태그를 이기는 것과 같은 계열. LLM 회차 흔들림은 주석 서술이 빈약해서지 계상 위치 자체가 불확실한 게 아니다(원문 표엔 매출원가/판관비에 명확). `표시위치_불안정_표구조우선` 기록. 표 구조로 확정 안 되는 불안정은 그대로 불명(코미코 매출채권 대손 등).
- **[1a] 프롬프트 보수 규칙 + [1b] 코드 근거 게이트 (`placement_basis_ok`)**: 프롬프트는 계상 위치를 드러낼 때만 상/하단 확정하고 그 **근거 구절을 `표시위치_근거`에 verbatim 으로 뽑게** 한다(surface/2.1, 능력 플래그 `placement_evidence_field`). 코드 게이트는 상단 확정에 대해 XBRL 확정이 아니고 `표시위치_근거`가 인용·주석위치에 verbatim 으로 **존재하지 않으면**(=지어낸/없는 근거) 불명으로 강등해 구역 B 진입을 막는다(`표시위치_강등`·`표시위치_llm` 기록, 헛숫자 차단과 같은 사후검증). **태그 내용을 재판정하는 게 아니라 근거 유무만 확인** — 계정명 목록 매칭 없음, LLM 이 지목한 근거 텍스트의 존재만 구조적으로 본다(no-keyword-heuristics 준수). 근거를 통과한 상단은 `표시위치_근거_확인`으로 원문보기에 띄운다. 하단은 EBITDA 조정과 무관해 게이트 대상 아님.
- **[1c] 표 구조 근거 인정 (`table_placement_ok`, 2026-07-15)**: 중소형사는 LLM 이 재고평가·명세 표에서 금액을 스티치해 인용을 만들면 verbatim 매칭이 깨져, 표엔 명백히 매출원가/판관비에 계상됐는데도 상단이 강등되거나 불명으로 태그된다. 표 구조 자체를 상단 근거로 인정한다 — **계정명 키워드가 아니라 표 산술 정합 + XBRL 개념코드로만**(XBRL 결정론[2]이 LLM 을 이기는 것과 같은 계열, no-keyword 준수). 근거 = 후보 금액이 ① 섹션 최대 당기셀이 손익계산서 매출원가/판관비/영업비용 라인과 표 산술로 정합하는 섹션, 또는 ② 재고평가 개념코드(`Writedown…Inventor`/`Reversal…Inventor`, IAS 2.34 매출원가 인식) 섹션에 문자열로 실재(`opex_notes.above_line_evidence_text`). **현금흐름표 섹션(`CashFlowsFromUsedIn` ACODE)은 제외** — CF 표엔 대손·충당부채 조정이 동거해 영업비용 표가 아니다. 세 경로: ⓐ 검증(LLM=상단·verbatim 실패 → 표 구조로 유지, `표시위치_근거_표구조`), ⓑ 결정(LLM=불명 → 표 구조로 상단 확정, `표시위치_표구조결정`), **ⓒ 불안정 우선(2026-07-15, LLM 회차마다 갈림 → 표 구조가 안정성 게이트[3]를 이겨 상단 확정, `표시위치_불안정_표구조우선`)**. **범위 제한(거짓 상단 0)**: 표 구조로 확정 안 되는 불안정은 그대로 불명(이마트 8·SK 5·삼성 4 불안정 승격 0), CF 대손은 근거에 없어 불명 유지(삼성 144,508·코미코 1,661,696). 검증(8사): 아이센스 878,525(ⓐ)·대한항공 8,485(ⓑ)·**코미코 666,343(ⓒ)** 구역 B 진입, 기존 B 전부 보존(전부 XBRL/verbatim 확정이라 표 구조 미개입), 불안정 이긴 항목은 코미코 2건뿐(666,343→B·전기 3,048,760→참고, 둘 다 IAS 2.34 재고평가)·거짓 상단 0, base 8사 불변. 재계산 공식 불변(`docs/limitations.md §13`).
- **[4] 근거강도 정의**: 명시=금액이 인용에 그대로 있어 값 직접 확인 / 추정=표 구조·문맥에서 유추. 추정 항목은 원문보기 발췌 범위를 넓혀(`window_scale=2`) 유추 근거를 함께 보인다. 근거강도는 표시용일 뿐 조정 가능 여부·재계산에 영향 없음.
- **불명의 방출 밸브 — 수동 조정 스위치(`render.py`)**: 근거 게이트·불명이 보수적이라 조정 기회를 놓칠 수 있으므로, 구역 C 불명 항목에 사람이 원문 확인 후 직접 조정에 넣는 스위치를 둔다. 손익방향 태그가 있으면 그 부호, 불명이면 사용자가 이익/비용을 고른다. 수동 포함은 **사용자 판단**으로 표시(도구 판단과 구분)되고 재계산 공식(base+Σsign×amount)의 같은 Σ에 항을 더한다 — 공식 불변. localStorage 없음(페이지 내 JS 상태).

## 기간 게이트 — 당기 금액만 조정 (2026-07-14)
구역 B(당기 EBITDA 조정)는 **당기 금액만** 들어간다. surface 후보는 주석 텍스트에서 LLM 이 읽은 것이라 기간 보장이 없어 전기(작년) 금액을 집을 수 있다(네이버 스톡그랜트 80,180=전기인데 상단 B 로 올라와, 토글하면 작년 숫자가 올해 EBITDA 에 반영). D&A·영업이익은 XBRL 개념코드(CFY{연도}dFY)로 뽑아 기간이 확실하지만 surface 후보는 아니다. `src/extract/period_notes.py` + build_view 게이트:
- 우선순위 **① XBRL 기간 컨텍스트**(`<TE ACONTEXT>` CFY{base_year}=당기/PFY=전기, 결정론·LLM 태그보다 우선) **> ② 인용 기간표현**(금액 앞 가장 가까운 당기/전기 — 계정명 아니라 기간어. '전기차·전기요금·전기전자' 합성어는 lookahead `전기(?=[\s중말초에:,()])` 로 제외) **> ③ LLM 기간 태그(전기만 신뢰해 강등) > 불명**. 당기 인정(B 허용)엔 XBRL·인용의 적극적 당기 근거 필요(표시위치처럼 보수·비대칭 — 모르면 불명).
- 구역 B 진입 조건에 `기간=당기` 추가. 당기 아니면(전기·불명) 조정 자격만 박탈 → 구역 C 불명(발굴 유지·수동 스위치 열어둠 — 리콜 우선). 재계산 공식 불변(게이팅만).

- **BS값 게이트(2026-07-15)**: 후보 금액이 재무상태표(sj_div=BS) 라인 잔액과 정확히 일치하면 손익 아닌 잔액 → 조정 대상 제외(자동·수동 불가, 구역 C 참고). `build_view.bs_amounts_won`. 표준 구분(계정명 키워드 아님). 예방적(6사 현재 발동 0·과제거 0). **한계**: 주석에만 있고 XBRL 개념코드 없는 장부금액(이마트 3,321,458=지분법투자, 순수 `<TD>`)은 못 잡음 — 키워드 없인 불가, 보수적 C불명에 둠(`docs §11`). **근거 게이트 비대칭 유지**: 하단은 무검증이나 하단은 "근거 확인" 배지 안 줘 오도 없고, 대칭화(하단→불명)는 확정 하단을 수동 스위치에 노출해 오히려 해로움(`docs §12`).

## 핵심 사냥감 — 매출원가·판관비에 묻힌 일회성
영업이익 아래 일회성(처분손익·손상)은 손익계산서에 별도 줄로 보여 눈에 띈다(참고 가치, 구역 C). 진짜 회색지대는 매출원가·판관비 안에 파묻힌 일회성(재고평가손익, 구조조정비, 충당부채 전입·환입, 일회성 수수료·보조금)이며, 상단(구역 B)이 그것을 캐내는 자리다. 하단을 스스로 "영업외라 뺀다"로 놓치지도 말되(빠짐없이 올린다), 분석 가치의 무게는 상단 발굴에 있다. 하단 발굴은 기존 방식 유지 — LLM이 주석을 읽어 구역 C 리스트를 올린다(결정론 알림 대체안은 채택 안 함).

## 범위 — in-scope / out-of-scope (골든셋·도구 공통 기준)
도구가 잡아야 하는 것 (in-scope):
1. 상단(영업이익 안) 비반복 → 조정대상(구역 B, 토글). 매출원가·판관비에 묻힌 일회성(재고평가손익·구조조정비·충당부채 전입환입·일회성 수수료/보조금). ← 이 도구의 핵심.
2. 하단(영업이익 아래) 일회성 → 참고(구역 C). 손상·처분손익 등, 손익계산서에 별도 줄로 보임.
3. 하단 규모 큰 손익 → 참고(구역 C). 금융평가이익·배당금 등 — 반복이라도 규모로 이익 해석에 영향을 주면 참고.
잡지 않아도 리콜 손실 아닌 것 (out-of-scope):
1. 추정품질 — 마일리지 소멸률·상각개시 판단 등 "숫자 신뢰성 의심"(금액 빼는 항목 아님). 손익 아닌 숫자(자산 규모 17.6조 등)로 리스트 오염. 이익질 일차 신호는 screen 괴리가 담당(`docs/limitations.md §4`).
2. 세금 항목 — 세액공제·글로벌최저한세 등. EBITDA(영업이익 기준선) 밖.
3. 자본거래·비손익 — 자기주식처분이익·자기주식 소각 등. 손익이 아니라 자본이 직접 변동.
경계: 위 세 out-of-scope 범주에 "명확히" 해당하는 것만 뺀다. 애매하면 in-scope 유지(리콜 보수적). 골든셋 재분류도 삭제가 아니라 `scope=out_of_scope` 표시(사람 승인 이력 보존, 리콜 분모에서 제외).

## 조정 표시 — 이중가산 방지 (표 산술 계층, EBITDA 를 움직이는 모든 경로에 적용)
- **탐지 = 표 산술(비율 폐기, 2026-07-12 교체)**: 같은 경제사건의 "합계"와 "구성"이 둘 다 조정에 들면 이중가산이다. [폐기] 옛 비율/임계(구성합이 합계의 99.5%+) 기준은 "구성요소가 전부 후보로 올라온다"는 성립 안 하는 전제 위에 섰다 — 작은 구성(계약손실환입 1,048)이 후보로 안 오르면 비율 미달로 못 잡고(충당부채 환입 35,092 ⊃ 판매보증환입 34,044=97%를 놓침), 임계를 풀면 금액만 가까운 남남(무형처분 38,663 vs 무형손상 38,072)을 과묶는다. **비율은 포함관계의 증거가 아니다 — 주석 표의 산술이 증거다.** `detect_containment(pool, notes)`: 주석 텍스트를 '숫자 런'(쉼표숫자가 공백·괄호로만 이어진 구간 — 한글 라벨이 끊음)으로 쪼개, 최댓값이 나머지의 **정확한 합**과 같은 런을 합계 행으로 본다. 매트릭스 표 행(라벨은 행 앞, 셀은 이웃)은 한 런이라 잡히고, 나열식 목록(값마다 라벨: `무형손상 38,072 무형처분 38,663`)은 값마다 런이 끊겨 합계로 오인 안 됨(과묶기 방지가 파서 구조에서 자연히 나온다). '합계'·'총액' 글자 안 찾음(키워드 금지) — 숫자가 실제로 더해지는지만. D&A '성격별 합계=유형+무형+리스' 산술 대조와 같은 기법.
- **계층 구성 + '그 외' 잔여(합=구성 항상 성립)**: 합계 확정 시 구성 후보를 계층 자식으로 달고, 표에서 이름 못 얻은 잔여(합계 − 확인된 구성 합)를 `computed_residual` '그 외' 노드로 채운다 — 원문 인용이 아니라 도구 계산값임을 명시(citations-mandatory: 출처 없는 숫자를 원문처럼 보이지 않게). SK: 충당부채 환입 35,092 = 판매보증환입 34,044 + **그 외 1,048**(계산값). 합=구성이 산술로 닫혀 이중가산이 산술적으로 불가.
- **계층 체크 규칙(3중 방어)**: ① 합계 체크 → 구성 전부 자동 체크(`propagateHierarchy`), ② 구성은 부모 합계가 켜지면 잠김(`locked`=구성만, 합계는 계층으로 안 잠김) → 체크박스 disabled, ③ Σ 게이팅(`!locked`) — 합계 체크 시 구성은 흡수돼 합계 1회만 Σ. 구성 개별 체크(합계 off)는 부분 조정 허용(합계 자동 체크 안 됨). 구성 전부 체크 = 합계값(이중 아님). 자동(B)·수동(C 불명) 양 경로 동일. SK 전수 시뮬 13,824조합·동시반영 0건, base EBITDA 61,095,958 불변.
- **탐지는 3단계 — 확신(산술)과 추측(근접)을 구분해 말한다(2026-07-12)**: 표 산술은 '같은 표 안'만 증명한다. 서로 다른 주석에 흩어진 포함관계는 산술 불가 → 이때만 금액 근접을 보조로 쓰되 추정임을 밝힌다(D&A 계상위치 안내·근거강도 명시/추정과 같은 정직성). **안전 기본값: 기본은 잠금(가만두면 이중가산 안 남), 푸는 건 사용자의 의식적 행동** — 실수의 방향이 안전 쪽으로 기울게.
  - **1단계 표 산술 확정** = 사실 → 잠금·계층·잔여. 경고·해제 불필요(증명됨). (위 두 bullet)
  - **2단계 고근접 추정**(`detect_containment_proximity`): 1단계로 확정 안 됨(다른 주석) + 근접 ≥99.5% + 동부호 + **성격 계열 동일**(기존 `성격` 태그 교집합, 계정명 매칭 금지) → 큰 값=합계·작은 값=구성으로 잠그되 `containment_estimated`로 **추정임을 표시**(✓확정과 구분)하고 [묶음 해제] 버튼(`unlinked` 셋·`linkActive`)으로 사용자가 풀 수 있다. 해제 시 두 항목 독립(이중가산 재경고). **잔여('그 외')는 안 만든다**(산술 근거 없어 계산 불가 — 1단계와 다른 점). SK: 지분법손상 471,006 vs Wuxi 470,869(주석 33 vs 11, 99.97%, 둘 다 비반복)=추정 잠금(옛 비율은 근거 없이 단정했으나 이제 추정으로 명시). **⚠ 한계**: 추정품질 제거 후 성격 태그가 `비반복` 하나뿐이라 성격 조건은 변별력이 약하다 — 실질 방어는 99.5% 임계 하나, 우연 근접한 별개 항목이 오잠금될 수 있음(세분은 발굴 위축 위험이라 미채택, 오잠금은 해제로 대응). `docs/limitations.md §4b`.
  - **3단계 중근접 경고**: 근접 90~99.5%(또는 ≥99.5%이나 성격 계열 달라 2단계 미충족)·동부호 → 잠그지 않고 `containment_warn` 배지만(남남 가능성 실질적, 과묶기 금지). 무형처분 38,663 vs 무형손상 38,072(98.47%)는 나열식(하단 참고)이라 pool 밖 + 임계 미달 → 안 걸림.
  - notes 없으면(`--no-context`) 1단계 불가 → 합계행 0, 근접(2·3단계)으로 방어(충당부채 97%는 3단계 경고, 지분법 99.97%는 2단계 추정).
- **후보 vs D&A 이중계상(별도 축, 2026-07-13)**: 위 1·2·3단계는 '후보끼리'만 본다 — '후보 vs D&A' 사각지대가 있었다. EBITDA=영업이익+D&A 라, D&A 로 가산한 상각 금액에 이미 포함된 항목(예: 고객관계 상각 ⊂ 무형자산상각비)이 조정 후보로 또 올라오면 체크 시 두 번 반영된다. `_da_double_count`: 후보 표시금액이 D&A 상각 셀(`all_da_cells_by_note` 를 XBRL 개념코드 `Depreciat|Amorti` 로 거른 것, 비영업 `InvestmentProperty|NotInUse` 제외)과 같고 같은 주석이면 조정 자격 박탈 → 참고. 개념코드 비교라 계정명 키워드 아님, 손상차손은 D&A 개념이 아니라 안 걸림(과제거 0 — 4사 검증: 대한항공 99,796 만 차단, 모든 손상·재고평가환입·주식보상 유지). **광역 보수 제외(D&A 구성 주석에서 온 건 다 제외)는 폐기** — 손상차손이 유형·무형 주석에 함께 공시돼 모든 손상을 과제거했다(리콜 붕괴). 정밀한 개념코드 셀 일치로 교체.
- **dedup 보강(2026-07-13)**: `_merge_key` 의 금액 병합키를 (금액, 첫 주석번호)→(금액, 주석번호 집합)으로 — 같은 금액·같은 주석 집합이면 이름·나열 순서 달라도 병합(대한항공 '고객관련/고객관계 무형자산상각비' 두 이름 병합, merged_count 2). tier-2 는 금액이 정확히 같으면(ratio==1.0) 포함관계가 아니라 중복이므로 제외(중복을 합계/구성으로 오분류하던 것 해소).
- **동일금액 이중가산 — 독립 안전망(2026-07-14, 병합 의존 붕괴 대응)**: 2차 스윕이 같은 항목(SK 재고평가환입 661,733)을 다른 주석에서 발굴하면 `_merge_key`(금액,주석집합)가 안 맞아 구역 B 에 같은 금액 2건이 남고, tier-2 는 ratio==1.0 을 "중복이니 병합이 처리"라며 제외해 **아무도 안 막는 사각지대**가 됐다(SK 100% 통과, 롯데 99.998% 는 잠긴 역설). **원칙: 방어는 다른 방어(병합)의 성공을 가정하면 안 된다.** 두 겹으로 해소(게이팅·병합만): (1-a) `_consolidate_same_amount_name` — 같은 금액·부호·정규화 항목명이면 주석 달라도 병합(이름 일치 요구=남남 오병합 방지, `_merged_locations`로 양쪽 인용 보존). (1-b) `detect_same_amount_locks` — 병합 성공과 **무관하게** 구역 B·수동경로에 금액 정확 동일 ≥2 면 상호배제 잠금(대표만 Σ, 나머지 게이팅, `sameAmtLocked`·[묶음 해제]). **계층 제외 범위(2026-07-15 교정)**: 계층 총계(is_total)와 '**1단계 산술 확정** 총계의 자식'만 제외한다 — 1단계 부모는 늘 켜져 자식이 확실히 잠기지만, **2단계 추정(`containment_estimated`) 총계의 자식은 부모를 사용자가 끌 수 있어(끄면 자식이 풀림) 계층 방어가 지킨다는 보장이 없다** → 그런 자식은 동일금액 그룹에 편입해 부모 상태와 무관하게 독립 상호배제(대표는 자립 항목 우선, 자식은 dup). 이 갭이 실제 뚫린 사례=이마트 165,697(대손 3변형 중 1건이 2단계 추정 총계 자식이라 빠져, 부모 끄면 이중가산 +331,394). "동일금액 잠금이 계층(다른 방어)의 성공을 가정한" 반복 실수를 교정. **부호 무관 그룹핑(2026-07-14)**: sign=None(손익방향 불명)도 잠근다 — 금액이 같으면 부호 불명이어도(사용자가 방향 지정 후 둘 다 켜는 경로 차단). 단 정부호가 상반(이익 vs 비용)이면 반대 항목이라 안 잠금(이마트 3,321,458 sign=None 두 변형 잠김). 검증: SK 병합 1건(−661,733 1회), 다른-이름 동일금액은 (1-b) 잠금(삼성2·네이버2·이마트3), 이마트 165,697 클러스터 16조합 전수 이중가산 0(추정 부모 끈 경로 포함)·8사 동일금액 잠금 수 불변(과잉잠금 0)·base 불변. `docs/limitations.md §8,§14`.
- 재계산 공식(base + Σ sign×amount)은 불변. 잠긴/박탈 항목이 Σ에서 빠지는 게이팅만 얹는다. 전수 시뮬: 기본(잠금) 13,824조합 동시반영 0, 2단계 해제는 사용자 의식적 노출(경고), 1단계는 해제 불가. 음수 base(영업손실) 에서도 부호 규약 동일(합성 테스트: 이익 체크→감소, 비용 체크→증가).
- 중복 병합 시 대표의 2차 태그 우선순위: 확정 > 불명(조정성격·손익방향·표시위치가 회차마다 갈리면 확정 쪽을 남긴다). 재현수는 그다음.

## 원문보기 — 검증 가능성 (citations-mandatory 확장)
- citations-mandatory 는 "인용이 진짜"에서 그치지 않는다. 원문보기는 사람이 **검증하는 도구**여야 한다 — ① 인용이 존재하고(없으면 안 됨), ② 표는 읽혀야 하고(줄바꿈), ③ 하이라이트가 실제로 작동해야 한다.
- 재무제표 라인(영업이익 등)은 XBRL 구조데이터로 손익계산서 계단(매출→매출원가→매출총이익→판관비→영업이익)을 표로 재구성해 띄우고 뽑은 줄을 하이라이트한다. "(인용 없음)"으로 두지 않는다. XBRL 표준 개념코드로 식별·정렬(회계기준 구조 = no-hardcoding 예외, 계정명 키워드 매칭 아님).
- 주석 표(원문보기)는 통째로 쏟지 않는다(방식 B): 구조 신호(쉼표숫자+공백=셀 끝)로 줄바꿈해 "항목명 금액" 한 행씩 읽히게 하고, 하이라이트 주변 행만 먼저 보이고 "전체 펼치기"로 나머지를 편다. 원문 훼손·요약 금지 — 발췌·줄바꿈만, 내용 불변.

## 철칙
도구는 "이게 일회성이다"를 판정하지 않는다. 후보를 빠짐없이 올리고(리콜), 판정은 감사인이 한다.
골든셋 판정 원칙도 "빼야 하나"가 아니라 "떠야 하나"다 — 애매하면 띄운다(리콜 기준).
리콜을 우선한다. 넓히면 정밀도가 떨어지는 건 정상 — 그 조절은 슬라이더와 사람이 감당한다.
